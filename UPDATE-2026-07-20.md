## Completed since the last update

**Deployment automation (Phases 4 and 9).** The system now starts with one command,
`./scripts/start.sh`, replacing an eight-step manual sequence. It provisions PostgreSQL in Docker,
generates configuration, applies migrations, seeds, and starts both services;
`./scripts/start.sh --fabric` additionally brings up the Fabric network, deploys the chaincode and
copies the gateway credentials.

**The two missing risk signals (Phase 6).** ROADMAP §4.1 specifies seven signals; two had never
been implemented. Both now exist and were confirmed firing during the evaluation:

- **Geovelocity** — a credential replayed from Sydney seconds after a London login was blocked at
  risk score 85. Geolocation is resolved from a local table rather than a commercial service,
  keeping the prototype self-contained per §8's ethics constraint.
- **Navigation sequence** — a twelve-endpoint discovery sweep raised the signal as designed.

**Lateral movement is now tested (Phase 8).** ROADMAP §1 names it as one of three security
challenges addressed, but none of the five original scenarios exercised it. A sixth scenario now
probes it from the strongest possible foothold — a fully legitimate, MFA-enrolled student session —
across three directions: reaching another student's records by injecting their identifier,
escalating to the administrative surfaces, and sweeping for undocumented endpoints. All 16 probes
per run were contained.

**On-chain identity revocation.** `IdentityContract.revokeIdentity` had no route into it from the
running system. It is now reachable through an administrator-only endpoint that revokes the anchor
on-chain, closes the student's live sessions, and records the act in the audit trail.

## Three defects found and fixed

Each was found by building or running the above, not by inspection.

1. **Blocked logins were not written to the audit trail.** When the engine refused a login
   outright, it returned 403 and recorded nothing. ROADMAP §4.2 defines DENY as "block this request
   *and* log it", so the most security-relevant event the system can produce was absent from the
   immutable record. Previously unreachable — adding geovelocity made that branch live, which is
   how it surfaced.

2. **Two database migrations were missing entirely.** Four schema objects — including the columns
   TOTP enrollment depends on — existed in the schema and in running databases but had never been
   captured in a migration. A fresh installation could not perform multi-factor authentication at
   all, and seeding crashed. Reproducing the system from the repository alone was therefore
   impossible; it now works from an empty database.

3. **The published API specification documented an endpoint that does not exist**, and described
   the identity check as using a method the code does not call. Anyone building against the
   documentation would have received a 404.

## Open item — needs your decision

**"Authentication Performance" (CES, 10% weight) still has no agreed definition.**

ROADMAP Table 1 assigns this component 10% of the composite score, but §7 — which gives explicit
formulas for the other three components — never defines it. No measurement can be final until it
is defined.

**Our proposal**, implemented and clearly flagged as provisional: measure mean login latency and
score it against published human-computer-interaction thresholds — full marks at or under
**3 seconds** (the widely used web-response threshold), zero at or over **10 seconds** (Nielsen's
*limit of attention*), interpolating between.

**The measured result is 3 310 ms, which slightly exceeds the 3-second target**, scoring 0.956 and
placing the fully weighted CES at 99.6 rather than 100.

We want to be explicit that this was left as measured. Raising the target to 3.5 seconds would
produce a perfect score, and we have deliberately not done so: a threshold selected after seeing
the result is not a threshold. Both anchors were taken from the literature before the evaluation
ran.

The shortfall is itself a finding. A synchronous on-chain identity check at login costs more than
the standard web-response threshold allows — which is precisely the trade-off quantified below, and
the evidence for our recommended next step.

**What we need from you:** confirm this definition, or supply your own acceptable login time.

## Measured cost of immutability

| | Off-chain | On-chain (Fabric) |
|---|---|---|
| Access decision | ~0.05 s | **~2.1 s** |
| Login (including anchor check) | 0.30 s | **3.31 s** |
| Storage per decision | 354 B | **~8 KB on every peer, permanently** |

An eleven-fold increase in login latency, attributable entirely to moving the identity check and
the decision write on-chain. The latency is endorsement by both organisations plus ordering and
commit; the storage is the transaction envelope, endorsement signatures and certificates that make
a record *provable* — the audit record itself is only 354 bytes of it.

**Interpretation.** Immutability is not free; it is paid for in latency and storage. The current
design suits high-value security events. For routine access at full university scale, periodic
Merkle-root anchoring — batching decisions and committing only the root — would preserve tamper
detection at a fraction of the cost. We recommend it as the next architectural step; it is outside
the scope of this prototype.

## Declared limitations

Stated plainly rather than left to be discovered, and all within ROADMAP §8's agreed scope.

- **Prototype scale.** Two organisations on a single host: a functional research deployment, not a
  production topology.
- **Small sample.** 50 labelled trials. Perfect scores across tens of trials demonstrate that the
  mechanisms work as designed; they are not a claim about behaviour at university scale.
- **Detection time is 7.19 s, not the 1.4 s** of the brief's illustrative table. The background
  monitor polls every 15 seconds, so mean detection cannot fall below roughly half that interval.
  It is a tuning constant traded against monitor load, not a limit of the approach.
- **Simplified signals.** Geolocation is table-driven; behaviour modelling is request rate and
  navigation breadth rather than behavioural biometrics. §8 already names these as future work.
- **Identity revocation is permanent** and deliberately manual — it is never triggered by a risk
  score, because a false positive must not be able to lock a student out of their records
  irreversibly.

All evaluation used synthetic data. No real student records or personal information were involved.
