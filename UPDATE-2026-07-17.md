# Progress Update — Hyperledger Fabric Integration

**Date:** 17 July 2026
**Project:** Blockchain-Enhanced Identity Verification for Zero Trust Access Control in University Student Portals

---

## Headline

**The system now runs on a live Hyperledger Fabric blockchain.** Until today the Zero Trust
engine wrote to a stand-in ledger while the Fabric work was completed. That final step is done:
every login and every access decision is now written to, and read back from, a real permissioned
blockchain — and the full evaluation has been re-run against it.

This completes ROADMAP Phases 1, 4 and 5, and re-validates Phases 6–9 on the real ledger.

## Two measured findings

Going live produced quantitative results a simulated ledger could not reveal. Both are useful
engineering conclusions rather than defects:

**1. Latency — ~2.1 s per access decision.**
Writing a decision to the blockchain requires endorsement by both organisations plus ordering
and commit. Login now takes ~2.6 s versus ~0.2 s off-chain. Concurrent requests queue, because
appends must be serialised to preserve the hash chain.

**2. Storage — ~8 KB per access decision, permanently, on every peer.**
The audit record itself is 354 bytes; the remaining ~7.6 KB is the transaction envelope,
endorsement signatures and certificates that make the record provable. Nothing can ever be
deleted.

**Interpretation.** Immutability is not free — it is paid for in latency and storage. The current
design suits high-value security events. For routine access at full university scale, periodic
Merkle-root anchoring (batching decisions and anchoring only the root) would preserve tamper
detection at a fraction of the cost. Recommended as the next architectural step.

## Also delivered today

- **Role-based access control.** The audit trail is now restricted to administrators
  (`STUDENT` / `ADMIN`). Previously any signed-in student could read every student's decisions.
  Staff and student roles are disjoint — neither is a super-user.
- **Four defects found and fixed**, none of which were reachable before going live:
  1. Chaincode state-iterator crash on a real peer (masked by the offline test stub).
  2. Audit verifier called with a student token — would have reported audit integrity as 0%.
  3. Database reset would permanently lock out every student against immutable anchors.
  4. An unreachable peer crashed the backend; it now returns a clean 503 and recovers.

## Open item for the client

**"Authentication Performance" (CES, 10% weight) still needs a definition.**

ROADMAP §7 introduces this component but does not define how to measure it. It is currently
scored provisionally against a 1500 ms placeholder budget, and CES is reported both including
and excluding it, so nothing is overstated.

We can now ask the question concretely: **measured login latency on the blockchain is ~2.6 s.**
No synchronous on-chain identity check will meet a 1500 ms budget. We need the client to confirm
what login latency is acceptable, so the final headline CES can be fixed.

CES excluding this component — the defensible figure today — is **100/100**.
